**DGrid x402 API** 是 DGrid 針對 API 付款所提供的按次推論付費存取模式。開發者與自主 Agent 不需要建立帳戶、預先儲值餘額，或管理獨立的計費流程，而是可以在 API 請求生命週期中，直接透過 **x402 支付協議** 支付 LLM 推論成本。

這讓 x402 特別適合以下情境：

* 需要呼叫 LLM 並立即完成付款的 **Agent 與自主工作流程**
* 應避免建立帳戶與手動儲值的 **伺服器端工具與腳本**
* 最簡單路徑就是「請求推論、授權付款、取得結果」的 **隨需整合**

## 為什麼 x402 API 很重要

傳統 AI 計費通常需要：

* 向供應商建立帳戶
* 預先儲值或綁定付款方式
* 管理 API 金鑰、配額與計費儀表板

在 DGrid 上使用 x402，流程會直接許多：

<Tip>
請求推論 → 接收付款要求 → 簽署一次 → 將付款與推論一起提交
</Tip>

對 Agent 系統而言，這是很大的簡化。決定要使用哪個模型的同一個執行環境，也能直接授權並結算該次推論呼叫的成本，而不必切換到獨立的帳戶管理流程。

## 主要使用情境：由 Agent 支付推論費用

x402 是為了讓 **Agent** 能完成以下流程而設計：

1. 選擇模型
2. 呼叫模型 API
3. 支付推論費用
4. 取得結果並繼續執行

這對以下情境特別有價值：

* 在後端工作流程中運作的 AI Agent
* 需要具備支付感知推論能力的 Web3 Agent 或協議
* 應將推論視為可直接支付資源的多步驟自動化

## 流程如何運作

DGrid 上的 x402 使用 **兩步驟請求流程**：

1. **首次請求不帶 `x-payment`**
   伺服器會回傳 `402 Payment Required`，並附上可簽署的付款要求。
2. **客戶端簽署後重試**
   客戶端會產生 `x-payment` 負載，並在第二次送出相同推論請求時加入付款標頭。

這種模式可讓付款授權保持明確，同時讓推論呼叫本身維持簡單且易於自動化。

## 支援的付款資產

目前的 x402 端點使用：

* **協議**：x402
* **網路**：`eip155:56`（BSC）
* **支付代幣**：`USD1`
* **資產合約**：`0x8d0D000Ee44948FC98c9B98A4FA4921476f08B0d`

## 營運說明

成功結算後，服務會在後端將付款紀錄寫入 `x402_payment_logs` 資料表。

## 何時使用 x402，何時使用 AI Gateway

如果你需要以下能力，請使用 **AI Gateway**：

* 傳統 API 金鑰存取方式
* 統一的帳戶式模型路由
* 採用標準計費流程的長期產品整合

如果你需要以下能力，請使用 **x402**：

* 逐次請求直接付款
* 無需預先儲值帳戶餘額
* 在單一流程中完成 Agent 友善的推論與付款
