<strong>DGrid x402 API</strong>는 API 결제를 위한 DGrid의 추론당 결제 액세스 방식입니다. 계정을 만들거나, 잔액을 미리 충전하거나, 별도의 과금 흐름을 관리하는 대신 개발자와 자율 에이전트는 API 요청 수명 주기의 일부로 <strong>x402 결제 프로토콜</strong>을 통해 LLM 추론 비용을 직접 결제할 수 있습니다.

이 방식은 특히 다음과 같은 경우에 유용합니다.

* LLM을 호출하고 즉시 결제를 완료해야 하는 **에이전트 및 자율 워크플로**
* 계정 발급과 수동 충전을 피해야 하는 **서버 측 도구 및 스크립트**
* 가장 단순한 경로가 "추론 요청 → 결제 승인 → 결과 수신"인 **온디맨드 통합**

## x402 API가 중요한 이유

기존 AI 과금 방식은 보통 다음을 요구합니다.

* 공급자 계정 생성
* 선불 잔액 충전 또는 결제 수단 연결
* API 키, 사용량 한도, 결제 대시보드 관리

DGrid에서 x402를 사용하면 흐름이 훨씬 직접적이 됩니다.

<Tip>
추론 요청 → 결제 요구사항 수신 → 한 번 서명 → 결제와 추론을 함께 제출
</Tip>

에이전트 시스템에서는 이것이 큰 단순화입니다. 어떤 모델을 쓸지 결정하는 동일한 런타임이 별도의 계정 관리 워크플로로 전환하지 않고도 그 추론 호출 비용을 승인하고 정산할 수 있습니다.

## 주요 사용 사례: 에이전트의 추론 비용 결제

x402는 <strong>에이전트</strong>가 다음을 수행해야 하는 경우를 위해 설계되었습니다.

1. 모델 선택
2. 모델 API 호출
3. 추론 수수료 결제
4. 결과를 받아 다음 실행으로 진행

다음과 같은 경우에 특히 가치가 있습니다.

* 백엔드 워크플로에서 동작하는 AI 에이전트
* 결제 인식 추론을 원하는 Web3 에이전트 또는 프로토콜
* 추론을 직접 결제 가능한 리소스로 다뤄야 하는 다단계 자동화

## 흐름 작동 방식

DGrid의 x402는 <strong>2단계 요청 흐름</strong>을 사용합니다.

1. **`x-payment` 없는 초기 요청**
   서버는 `402 Payment Required`와 함께 서명 가능한 결제 요구사항을 반환합니다.
2. **클라이언트 서명 후 재시도**
   클라이언트가 `x-payment` 페이로드를 생성하고 동일한 추론 요청을 결제 헤더와 함께 다시 보냅니다.

이 패턴은 결제 승인을 명시적으로 유지하면서도 추론 호출 자체는 단순하고 자동화하기 쉬운 상태로 유지합니다.

## 지원 결제 자산

현재 x402 엔드포인트는 다음을 사용합니다.

* **프로토콜**: x402
* **네트워크**: `eip155:56` (BSC)
* **결제 토큰**: `USD1`
* **자산 컨트랙트**: `0x8d0D000Ee44948FC98c9B98A4FA4921476f08B0d`

## 운영 참고

정산이 성공하면 서비스는 백엔드의 `x402_payment_logs` 테이블에 결제 로그를 기록합니다.

## x402와 AI Gateway를 언제 써야 하나요?

다음이 필요하다면 <strong>AI Gateway</strong>를 사용하세요.

* 전통적인 API 키 접근
* 통합 계정 기반 모델 라우팅
* 표준 과금 워크플로를 갖춘 장기 제품 통합

다음이 필요하다면 <strong>x402</strong>를 사용하세요.

* 요청별 직접 결제
* 미리 충전한 계정 잔액 없음
* 하나의 흐름에서 에이전트 친화적 추론과 결제
